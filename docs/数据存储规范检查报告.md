# 接口数据存储规范检查报告

## 检查日期

## 项目架构概览

### 后端存储系统
- **MongoDB 为主**：当配置 MONGODB_URI 环境变量时，优先使用 MongoDB
- **本地文件 fallback**：未配置 MongoDB 时，使用本地 JSON 文件存储
- **client-state-repository**：专门的客户端状态管理模块

### MongoDB 集合设计
- `users`：用户核心信息（积分、会员状态等）
- `client_devices`：设备关联记录
- `tasks`：任务/处理记录
- `points_records`：积分变动记录
- `orders`：订单记录
- `user_preferences`：用户偏好（收藏、最近使用）
- `client_states`：完整客户端状态快照

## 发现的问题

### 问题 1：支付订单存在内存中
**位置**：backend/server.js 第 1073-1137 行
**问题描述**：
```javascript
const pendingOrders = new Map();
```
订单只存在于内存中，重启会丢失

**建议修复**：
- 订单应持久化到 MongoDB 的 orders 集合
- 或使用 client-state-repository 的订单存储功能

### 问题 2：前端同步逻辑被禁用
**位置**：utils/task-store.js
**问题描述**：
为避免本地数据被覆盖，临时禁用了 `applyRemoteState` 的用户状态同步

**建议修复**：
- 启用云同步，但正确处理本地和远程数据优先级
- 使用 `preferRemote` 参数控制同步方向

### 问题 3：payment.js 中直接操作本地存储
**位置**：services/payment.js
**问题描述**：
支付成功后直接写 `wx.setStorageSync`，但没有正确触发云同步

**建议修复**：
- 使用 task-store 的标准方法更新状态
- 确保每次状态变更都会标记为需要同步

## 推荐数据存储标准

### ✅ 应存储在数据库的数据
1. 用户账户信息（积分、会员）
2. 所有订单记录
3. 积分变动历史
4. 任务/处理记录
5. 用户偏好（收藏、最近使用）

### ⚠️ 可临时本地存储的数据
1. UI 状态缓存
2. 离线临时记录（待同步时上传）
3. 图片/文件本地缓存

### ❌ 不应仅存本地的关键数据
1. 用户余额
2. 会员状态
3. 订单信息
4. 交易记录

## 风险评估

| 风险项 | 严重度 | 说明 |
|--------|--------|------|
| 订单内存存储 | 🔴 高 | 服务重启会丢失数据 |
| 云同步被禁用 | 🟡 中 | 跨设备数据不一致 |
| 本地存储依赖 | 🟡 中 | 小程序数据清理会丢失 |

## 修复优先级

1. **高优先级**：订单持久化到数据库
2. **中优先级**：修复云同步逻辑，确保数据正确双向同步
3. **低优先级**：完善事务和重试机制
